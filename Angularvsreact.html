Angular or React

In this article I want to tell my experience of working with Angular for a while and my change to React a few months ago. It is not an "Angular vs React" post, both of them seem good technologies, if I do not just want to show what things caused that I left Angular and tried different stack for the Front-End.

At that time, Node.js also began to hit hard and the so-called MEAN stack was formed (with MongoDB, ExpressJS, Angular and Node.js).

What was there before Angular

Angular is great, because at the time, the "good" alternative was Backbone and the "bad" JQuery.

A few years ago there was not as much business logic in the Front-End side of web applications as now, but if you wanted to do something, you could do it in pure JavaScript or help you from libraries like jQuery that reduced the cross-browser code and helped you manipulate the DOM.

The bad thing about jQuery was that in the end you forgot to program in JavaScript and you ended up using jQuery for everything

Backbone solved this by implementing an MVC architecture for the Front-End. The problem is that JavaScript a few years ago did not have the power or the tools that we have now. So if you wanted to modularize your code, you had to use Require.js, which, in my opinion, was a pain.

The arrival of Angular.js

Then appeared Angular, which at the time was a way to extend the HTML through directives. It also provided a MVC pattern, called MVVM (Model View-View Model), implemented double data-binding, management of modules, promises, etc ... Many things that were not in JavaScript at that time and that we wanted to use jQuery or JavaScript Vanilla to do it, it would be a lot of code.

The problem came when JavaScript evolved and the arrival of the new standard (ES6) added natively functionalities that Angular implemented.

We now have module management natively (it is not currently supported but it is there). We also have Promises natively, the Fetch API, etc ... A lot of new features that we do not need any library or framework to help us (Except Babel to transpile code).

If you want to use ES6 in Angular 1x, things get pretty complicated. Angular has its own system of modules and if you want to use ES6 (or use require with Browserify) you have to put almost every controller in an Angular module, and then export it with ES6.

If ES6 makes your life easier when programming in JavaScript, joining it with Angular made it more complicated

The birth of the WebComponents and Angular 2

Parallel to the development of Angular, the specification of the WebComponents was proposed and it was the beginning of the programming oriented to components. Polymer and React followed that architecture and in Angular 1x it could be achieved, using directives as components. In the end, an Angular directive is a component (it has an HTML tag) and encapsulates an HTML template and a JavaScript controller.

In fact, as of version 1.5, the component () function appears, which allows us to do just that, transform a directive into a component.

Seeing how JavaScript and the Web Components standard evolve, it was proposed to rewrite Angular following the new standards, is what is known as Angular v2 or simply Angular.

The problem is that, in my opinion, the syntax was very complicated. The new version of Angular was nothing like the previous one. Attempts were made to make the transition easier (Developing versions 1.5, 1.6 and 1.7 that were incorporating the novelties of Angular 2 gradually).

During all that time, React was one of the libraries that most took off and began to mature and its ecosystem and community grew a lot. It started to become a stable project and since I gave it a look some time ago, I wanted to give it a try

React is not the substitute for Angular, because React is just a library that does only one thing: Render components in the DOM. But he does it very well. To implement with React and its ecosystem, a web application as we would with Angular, we need more libraries and more programming.

This may seem like a disadvantage to some people, but to me personally it seems the opposite. With a framework, either Angular or any other. If you want to use for example a routing system, you have to use the one that is built for the framework itself.

With React, if you want to implement Routing, you can do it yourself with JavaScript Vanilla, or use a library, whichever you prefer. It can be the one that created the community with react-router, or you can use page.js or whichever you prefer. You are not tied to anything.

React allows you to be more modular and add whatever you need little by little. If you need to use promises you can use the ones that ES6 provides you in a native way, if you need to make AJAX calls, you can use fetch or a library as request or axios or even jQuery are $ .ajax ().

conclusion

This form is the one that seems most correct, because it allows you to be more independent and the architecture of your application no longer depends on a single library or tools. Also programming in this way has made me learn more about front-end architectures and improve as a professional. As I always say, this is my opinion, based on my experience with both technologies. Angular 2 is an excellent framework (if not, nobody would use it) and React is another technology within the great ecosystem that exists in the Frontend (Polymer, Vue, Backbone, etc ...)
